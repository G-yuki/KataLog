## 1. アーキテクチャと課金ポイント

```
ユーザー操作
  │
  ├─ アイテム一覧（ホーム）
  │    └─ カード表示 ──→ Storage URL（課金なし）
  │
  ├─ アイテム詳細を開く（placeId===null のとき初回のみ）
  │    └─ enrichItem（Cloud Function）★2ステップ実装済み
  │         ├─ [Places Text Search Essentials] ← ✅ places.photos 除去済（2026-04-29）
  │         │    $0.00283/req・10,000/月無料
  │         └─ [Place Details Photos Enterprise] ← placeIdが取れた場合のみ
  │              $0.007/req・1,000/月無料
  │              ※ 詳細を開かない限り API 不使用（lazy）
  │
  ├─ 詳細画面で写真を表示（placePhotoRefが参照名のとき）
  │    └─ Place Photo Media API（ブラウザから直接）← Enterprise $0.007/枚
  │         ブラウザキャッシュにより同一デバイスでは初回のみ課金
  │
  ├─ URL保存（詳細画面・前回と異なるURLのとき）  ← ✅ 同一URL再保存はスキップ
  │    └─ enrichItem（Cloud Function） ← 上記と同じ API
  │
  ├─ SUGGEST ──→ generateItems（Cloud Function）
  │               └─ [Gemini API]
  │
  └─ MEMORY ──→ generateMemory（Cloud Function）
                 └─ [Gemini API]

【tryアイテムのライフサイクル】
  スワイプマッチング → matchTier="try" + expireAt（+30日）をセット
           → 詳細を開いた場合のみ lazy に enrichItem 呼び出し
           → 30日後: Firestore TTL が自動削除（write quota カウントなし・無料）
           → onDocumentDeleted トリガーが Storage 写真を削除
```

## 7. tryアイテム自動削除（Firestore TTL）

### 仕組み

```
1. スワイプマッチング後 matchTier="try" のアイテム作成時：
   expireAt: Timestamp.fromDate(new Date(Date.now() + 30 * 24 * 60 * 60 * 1000))
   を Firestore ドキュメントに追加する

2. Firebase コンソール > Firestore > TTL ポリシー を設定：
   コレクショングループ: items
   フィールドパス: expireAt

3. Firestore が expireAt を過ぎたドキュメントを自動削除（通常24時間以内）
   → write quota カウントなし・無料
   → onDocumentDeleted トリガーが発火 → Storage 写真を削除
```

> `enrichPairItems` CF は削除済み。try アイテムは詳細画面を開いたときに `enrichItem` で lazy enrich される。

### 8-B. Storage ダウンロードコスト（写真表示分）

写真を画面に表示する際、ブラウザが Storage からデータを取得するため Firebase の「ダウンロード」として課金対象になる。
lazy 設計（最初の3枚のみ自動ロード）により従来の全件ロードより大幅に抑制される。

**月間ダウンロード量 / ペア：**
1.5回/日 × 30日 × 0.9MB = **40.5MB（0.04GB）/ ペア / 月**
